## 启动流程

Lite 框架由 `ModuleRunner` 驱动，按以下顺序拉起所有模块：

```text
1. ModuleRunner::Run()
       ↓
2. 创建 LiteClientBase（持有 Hub 的引用）
       ↓
3. REGISTER_LITE_MODULE 注册的工厂函数创建 Module 实例
   Module 构造函数：加载配置、初始化 CommunicationManager 等
       ↓
4. Module::OnInit()
   初始化成员变量、创建子组件（如 LocalVehicleImpl / CockpitImpl）
       ↓
5. Module::OnSubscribeChannels()
   调 Subscribe() 向 Hub 注册回调（含委托给 Component 的订阅）
       ↓
6. Module::OnSetUpTimers()
   向 Runner 注册周期任务（1Hz / 10Hz / 50Hz / 100Hz 等）
       ↓
7. Runner 进入事件循环
   ├─ 消息到达 → Hub 派发 → Module/Component 的 Handle* 回调
   └─ 定时器触发 → 对应 lambda / 成员函数执行
       ↓
8. 收到停止信号
   Module::OnModuleNotifyStop() → Stop() → 析构

## 分支管理

| 分支 | 用途 |
|------|------|
| `main` | 稳定主干，保护分支，不直接提交 |
| `develop` | 集成分支，各 feature 合入此处 |
| `feature/*` | 开发新功能，如 `feature/add-rtc-timeout` |
| `bugfix/*` | 修复 bug，如 `bugfix/fix-chassis-crash` |

```bash
git checkout -b feature/my-feature   # 创建并切换到新分支
git branch -d feature/my-feature     # 删除本地分支
git branch -a                        # 查看所有分支（含远程）
```

---

## 冲突解决

### git merge — 合并分支（保留完整历史）

```bash
git checkout main
git merge feature/my-feature   # 把 feature 合入 main
```

结果：生成一个**合并提交**，历史是有分叉的。

### git rebase — 变基（线性历史）

```bash
git checkout feature/my-feature
git rebase main    # 把 feature 的提交接到最新 main 之后
```

结果：历史变成**一条直线**，提交记录更干净。

**必须知道的区别**：

| | merge | rebase |
|--|-------|--------|
| 历史形状 | 有分叉（真实） | 线性（整洁） |
| 适合场景 | 合入主干（保留分叉记录） | 整理自己的 feature 分支 |
| 风险 | 低 | **已推送的分支不能 rebase**（会改写历史） |

冲突发生时都需要手动编辑冲突文件，然后：
```bash
git add <resolved-file>
git merge --continue   # 或 git rebase --continue
```

# 七、Docker

## 为什么用Docker

统一开发环境，避免"在我机器上能跑"的问题：

```text
库版本不同   → 容器内固定依赖版本
依赖缺失     → 镜像包含所有依赖
环境不一致   → 所有人用同一个镜像
```

---

## 核心概念

### 镜像（Image）

**只读模板**，包含操作系统 + 依赖 + 代码。类比：可执行文件。

```bash
docker images                  # 查看本地所有镜像
docker pull ubuntu:22.04       # 从 Docker Hub 拉取镜像
docker rmi <image_id>          # 删除镜像
```

---

### 容器（Container）

**镜像的运行实例**，可以启动、停止、删除。类比：进程。

同一个镜像可以启动多个容器，互相隔离。

---

### Dockerfile

描述如何构建镜像的脚本：

```dockerfile
FROM ubuntu:22.04              # 基础镜像
RUN apt-get install -y python3 # 安装依赖
COPY . /app                    # 复制代码
CMD ["python3", "/app/main.py"]# 启动命令
```

---

## 常用命令

### docker build — 根据 Dockerfile 构建镜像

```bash
docker build -t my-image:v1 .         # 在当前目录构建，打 tag
docker build -f path/Dockerfile .     # 指定 Dockerfile 路径
```

---

### docker run — 启动容器

```bash
docker run ubuntu:22.04 echo "hello"        # 运行一次性命令
docker run -it ubuntu:22.04 bash            # 交互模式进入容器
docker run -d --name my-app my-image:v1     # 后台运行，命名容器
docker run -v /host/path:/container/path \  # 挂载本地目录
           -p 8080:80 my-image:v1           # 端口映射 宿主:容器
```

---

### docker ps — 查看运行中的容器

```bash
docker ps           # 查看正在运行的容器
docker ps -a        # 查看所有容器（含已停止的）
```

---

### docker exec — 进入运行中的容器执行命令

```bash
docker exec -it my-app bash         # 进入容器的 bash shell
docker exec my-app ls /app          # 在容器内执行单条命令
```

和 `docker run` 的区别：`exec` 是进入**已在运行**的容器，`run` 是启动新容器。

---

### docker logs — 查看容器日志

```bash
docker logs my-app              # 查看全部日志
docker logs -f my-app           # 实时追踪日志（类似 tail -f）
docker logs --tail 100 my-app   # 只看最后 100 行
```

# 八、Bazel

## 为什么自动驾驶喜欢Bazel

| 特点 | 说明 |
|------|------|
| 项目规模大 | 数千个 target，Make/CMake 难以管理 |
| 依赖复杂 | 精确声明依赖，不会多编也不会漏编 |
| 增量编译 | 只重新编译真正变化的文件，节省大量时间 |
| 远程缓存 | 同事编译过的结果可以直接复用（CI 加速） |
| 跨平台 | 同一套规则编译 x86 / J6 / Orin 不同目标 |

---

## 核心文件

### WORKSPACE

放在仓库根目录，声明**外部依赖**（第三方库从哪下载）：

```python
# WORKSPACE
http_archive(
    name = "com_google_protobuf",
    urls = ["https://github.com/protocolbuffers/protobuf/..."],
)
```

---

### BUILD

放在每个目录下，定义这个目录里的**编译目标（Target）**。

#### 最简单的 BUILD 结构

```python
# onboard/my_module/BUILD

# 1. 编译一个库（其他 target 可以依赖它）
cc_library(
    name = "my_lib",              # target 名称
    srcs = ["my_lib.cc"],         # 源文件
    hdrs = ["my_lib.h"],          # 头文件
    deps = [                      # 依赖的其他 target
        "//onboard/proto:chassis_cc_proto",
        "@com_google_absl//absl/status",
    ],
    visibility = ["//visibility:public"],  # 允许被其他目录依赖
)

# 2. 编译一个可执行文件
cc_binary(
    name = "my_main",
    srcs = ["main.cc"],
    deps = [":my_lib"],           # 依赖同目录的 my_lib
)

# 3. 编译一个测试
cc_test(
    name = "my_lib_test",
    srcs = ["my_lib_test.cc"],
    deps = [
        ":my_lib",
        "@com_google_googletest//:gtest_main",
    ],
)
```

#### Target 地址格式

```text
//onboard/remote_drive/remote_drive_control:remote_control_module
  ↑                                          ↑
  从仓库根开始的路径                          BUILD 里的 name
```

同目录内可以简写为 `:my_lib`。
